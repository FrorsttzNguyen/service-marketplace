OOP & Design Patterns

Object-Oriented Programming, SOLID, Design Patterns, JPA — tất cả qua project Service Marketplace

1. OOP là gì? — Object-Oriented Programming

OOP = cách tổ chức code thành “objects” (đối tượng) giống đời thực.

Thay vì viết code lộn xộn, mỗi object “đại diện” cho một thứ thật — có dữ liệu riêng và hành vi riêng.

Ví dụ đời thực

👤

User object

Dữ liệu (fields): tên, email, password

Hành vi (methods): đăng nhập, đổi mật khẩu, cập nhật profile

📅

Booking object

Dữ liệu (fields): ngày giờ, dịch vụ, trạng thái

Hành vi (methods): confirm, cancel, hoàn thành

💰

Money object

Dữ liệu (fields): số tiền (cents)

Hành vi (methods): cộng, trừ, nhân

Key idea: Mỗi object = một “viên gạch” độc lập. Ghép các viên gạch lại → xây được cả hệ thống.

2. Encapsulation (Đóng gói)

Nguyên tắc: private fields + public methods = dữ liệu được bảo vệ.

Không ai sửa trực tiếp dữ liệu bên trong — phải đi qua methods có validation.

Ví dụ: Money Value Object

public class Money {
    private final long amountCents; // ← private: không ai sửa trực tiếp

    public Money(long amountCents) {
        if (amountCents < 0) {
            throw new IllegalArgumentException("Money cannot be negative");
        }
        this.amountCents = amountCents; // ← validation trong constructor
    }

    public Money add(Money other) {
        return new Money(this.amountCents + other.amountCents); // ← trả về object mới (immutable)
    }

    public long getAmountCents() {
        return amountCents; // ← read-only, không có setter
    }
}
Tại sao không có setter? Vì Money là immutable (bất biến). Khi cần thay đổi, tạo object mới thay vì sửa object cũ. Điều này ngăn bugs từ dữ liệu không hợp lệ và an toàn cho multi-thread.

Hình ảnh minh họa: Encapsulation giống "ngân hàng"

Private Field
🔒
amountCents = 50000
Tiền trong két sắt — không ai đụng được
Chỉ truy cập qua →
Public Methods
💼
getAmountCents() → đọc
add(other) → tạo mới
Constructor → validate
Nhân viên ngân hàng — kiểm tra trước khi giao dịch
Tại sao quan trọng: Encapsulation ngăn bugs từ invalid data. Nếu ai đó truyền -100 làm số tiền → constructor reject ngay. Không có cách nào tạo Money âm.

3. Inheritance vs Composition

Inheritance (Kế thừa)

Quan hệ "IS-A"

Dog is an Animal. Car is a Vehicle.

Class con LÀ phiên bản đặc biệt của class cha.

Composition (Phức hợp)

Quan hệ "HAS-A"

Car has an Engine. Vendor has a User.

Object chứa object khác như thành phần.

Trong project: Vendor và User

// ❌ KHÔNG dùng: Inheritance (Vendor IS-A User — không linh hoạt)
public class Vendor extends User {
    private String businessName;
    // Vấn đề: Vendor LÀ User mãi mãi. Nếu user muốn trở thành vendor?
    // Nếu một người vừa là customer vừa là vendor? → Khó xử lý.
}

// ✅ DÙNG: Composition (Vendor HAS-A User — linh hoạt hơn)
@Entity
public class Vendor {
    @ManyToOne
    private User user;  // ← Vendor CÓ MỘT User, không phải LÀ User
    private String businessName;
    // User có thể là customer và vendor cùng lúc!
    // Xoá vendor không ảnh hưởng user.
}

Minh họa: Inheritance vs Composition

Inheritance (IS-A)
User
email, password
extends ↓
Vendor
+ businessName
❌ Khó đổi role
Composition (HAS-A)
User
email, password
has-a →
Vendor
businessName, user
✅ Linh hoạt
Nguyên tắc: "Favor composition over inheritance" — Ưu tiên Composition. Inheritance chỉ dùng khi chắc chắn quan hệ IS-A là vĩnh viễn.

4. Polymorphism (Đa hình)

Đa hình = cùng một tên method, nhưng hành vi khác nhau tù vào loại object.

Giống như nút "Play" — trên TV là play video, trên Spotify là play nhạc, trên game là bắt đầu.

Ví dụ: PricingType Strategy

Trong Service Marketplace, mỗi dịch vụ có cách tính giá khác nhau. Dùng enum với abstract method để hiện thực polymorphism:

public enum PricingType {
    FIXED {
        @Override
        public Money calculatePrice(Money basePrice, int durationMinutes) {
            return basePrice; // Giá cố định, bất kể thời gian
        }
    },
    HOURLY {
        @Override
        public Money calculatePrice(Money basePrice, int durationMinutes) {
            // basePrice = giá mỗi giờ, tính theo thời gian thực tế
            int hours = (durationMinutes + 59) / 60; // làm tròn lên
            return basePrice.multiply(hours);
        }
    };

    public abstract Money calculatePrice(Money basePrice, int durationMinutes);
}

Cùng method call, kết quả khác nhau

Cùng method call:
pricingType.calculatePrice(basePrice, 90)
FIXED
↓
Trả về basePrice
Giá không đổi
HOURLY
↓
Trả về basePrice × 2
90 phút = 2 giờ
Lợi ích: Code gọi method không cần biết loại PricingType nào. Nó chỉ gọi calculatePrice() và kết quả tự động đúng. Thêm loại giá mới? Chỉ cần thêm enum value.

5. SOLID Principles — với project examples

SOLID = 5 nguyên tắc giúp code dễ bảo trì, mở rộng, và ít bug. Không phải luôn phải tuân thủ 100%, nhưng hiểu chúng giúp viết code tốt hơn.

S

Single Responsibility Principle

Mỗi class chỉ có MỘT nhiệm vụ.

// ❌ BAD: "God class" làm hết mọi thứ
class UserServiceBookingPayment {
    void registerUser() { ... }
    void createBooking() { ... }
    void processPayment() { ... }
    // Sửa payment → có thể làm hỏng user registration. BAD.
}

// ✅ GOOD: Mỗi class một nhiệm vụ
class User { ... }        // Quản lý user data
class Booking { ... }     // Quản lý booking lifecycle
class Payment { ... }     // Quản lý payment processing
Trong project: User.java chỉ handle user data. Booking.java chỉ handle booking logic. Payment.java chỉ handle payment. Sửa một thứ không ảnh hưởng thứ khác.
O

Open/Closed Principle

Mở để mở rộng, đóng để sửa đổi. Thêm tính năng = thêm code mới, không sửa code cũ.

Hiện tại
PricingType: FIXED, HOURLY
→
Thêm mới
PricingType: + PACKAGE

Chỉ cần thêm PACKAGE { ... } vào enum. Không cần sửa code của FIXED hay HOURLY. Code cũ hoạt động y hệt.

L

Liskov Substitution Principle

Các subtype phải thay thế được base type. Dù ở đâu, cứ gọi calculatePrice() là phải hoạt động.

Bất kỳ PricingType nào (FIXED, HOURLY, PACKAGE...) đều có thể dùng ở bất kỳ nơi nào cần PricingType. Không có "ngoại lệ" hay "case đặc biệt".

I

Interface Segregation Principle

Không ép class implement interface mà nó không dùng. Nhiều interface nhỏ > một interface to.

❌ Fat interface
interface BookingPaymentReview { book(); pay(); review(); }

Customer phải implement pay() và review() cả khi chưa cần.

✅ Small interfaces
interface Bookable { book(); }
interface Payable { pay(); }
interface Reviewable { review(); }

Mỗi class chỉ implement những gì nó cần.

D

Dependency Inversion Principle

Phụ thuộc vào abstractions (interface), không phải concrete classes.

// Domain layer định nghĩa interface
public interface BookingRepository extends JpaRepository<Booking, Long> {
    // Domain chỉ biết: có thể lưu và tìm booking
    // KHÔNG biết: dùng PostgreSQL hay MySQL hay in-memory
}

// Infrastructure layer cung cấp implementation (Spring Data JPA tự tạo)
// → Domain không phụ thuộc vào database cụ thể
// → Test có thể dùng mock repository
Domain Layer BookingRepository (interface)
implements ↑
Infrastructure Layer JpaRepository implementation (auto-generated)

6. Design Patterns trong project

Design Patterns = giải đà được kiểm chứng cho các vấn đề thường gặp. Không phải "silver bullet" — chỉ dùng khi thêm rõ ràng.

Pattern State Machine Pattern

Quản lý vòng đời của object thông qua các trạng thái và chuyển đổi hợp lệ.

↓
cancel

Implementation với Enum

public enum BookingStatus {
    PENDING, CONFIRMED, IN_PROGRESS, COMPLETED, CANCELLED;

    // Định nghĩa các chuyển đổi được phép
    private static final Map<BookingStatus, Set<BookingStatus>> TRANSITIONS = Map.of(
        PENDING, Set.of(CONFIRMED, CANCELLED),
        CONFIRMED, Set.of(IN_PROGRESS, CANCELLED),
        IN_PROGRESS, Set.of(COMPLETED)
    );

    public boolean canTransitionTo(BookingStatus target) {
        return TRANSITIONS.getOrDefault(this, Set.of()).contains(target);
    }
}
Tại sao? Không có state machine → ai cũng có thể chuyển COMPLETED → PENDING (vô lý). Có state machine → chỉ những chuyển đổi hợp lệ mới được phép.

Pattern Strategy Pattern

Nhiều thuật toán khác nhau đằng sau cùng một interface. Chọn thuật toán tại runtime.

Đây chính là phần PricingType ở mục Polymorphism. Strategy Pattern và Polymorphism đi cặp với nhau.

Lợi ích: Thêm listener mới (VD: SMSListener) không cần sửa code của Booking. Chỉ đăng ký (subscribe) là xong.

Pattern Value Object Pattern

Object không có identity, so sánh bằng giá trị, bất biến (immutable).

Entity

Identity: Có id
Mutable: Có thể thay đổi
So sánh bằng: id
VD: User, Booking, Order
Hai User cùng tên = cùng người? → Không, khác id.

Value Object

Identity: Không có id
Mutable: Immutable (bất biến)
So sánh bằng: Giá trị
VD: Money, Address, TimeSlot
Hai Money 50000 cents = giống nhau? → Có, giá trị bằng nhau.
Tại sao immutable? Thread-safe (nhiều thread đọc cùng lúc không sợ conflict), no side effects (không ai sửa giấu), predictable (luôn biết giá trị).

7. Domain-Driven Design (DDD) Basics

DDD = cách thiết kế software sao cho code phản ánh chính xác thế giới thực (domain). Thuật ngữ trong code = thuật ngữ business.

Entity

Có identity (id), mutable

User với id=1 và id=2 là hai người khác nhau, dù cùng tên.

User, Booking, Order, Payment, Vendor

Value Object

Không identity, immutable

Money(50000) và Money(50000) là một. So sánh bằng giá trị.

Money, Address, TimeSlot, PhoneNumber

Aggregate

Nhóm entities treated as one unit

Order là aggregate root. Thao tác với Payment và Refund phải đi qua Order.

Order (root) → Payment → Refund

Aggregate Boundary Diagram

id, status, total
Entity

Entity
Refund
amount, reason, status
Rule: Bên ngoài chỉ giao tiếp với Aggregate Root (Order). Không truy cập trực tiếp Payment hay Refund.

8. JPA Annotations Tour

JPA Annotations = cách nói với Spring/Hibernate "map Java class này vào database table nào, column nào". Mỗi annotation là một "chỉ dẫn" cho ORM.

@Entity
Map class → database table. Class nào có @Entity sẽ có table tương ứng trong DB.
class User { ... }
→
CREATE TABLE user (...)
@Table(name="users")
Tùy chỉnh tên table. Mặc định JPA dùng tên class, nhưng thường muốn tên table số nhiều hoặc khác.
@Table(name="users") class User
→
Table: users (không phải "user")
@Id + @GeneratedValue
Khóa chính (primary key) với tự động tăng. Mỗi row có một id duy nhất.
@Id @GeneratedValue Long id
→
id BIGSERIAL PRIMARY KEY
@Column(nullable, unique, length)
Tùy chỉnh column: có được để trống không, có duy nhất không, độ dài bao nhiêu.
@Column(nullable = false, unique = true, length = 255)
private String email;
// → email VARCHAR(255) NOT NULL UNIQUE
@OneToMany / @ManyToOne
Quan hệ một-nhiều. Vendor có nhiều Services. Service thuộc về một Vendor.
Vendor (One)
@OneToMany List<Service> services
↔
Service (Many)
@ManyToOne Vendor vendor
@ManyToMany + @JoinTable
Quan hệ nhiều-nhiều qua bảng trung gian. Vendor thuộc nhiều Categories, Category có nhiều Vendors.
vendors
(id, name...)
vendor_categories
(vendor_id, category_id)
categories
(id, name...)
@Embedded / @Embeddable
Value Object lưu trong cùng table. Address không có table riêng — các fields của Address được lưu thẳng vào table Vendor.
Vendor table
id, business_name
street (từ Address)
city (từ Address)
zip_code (từ Address)
@Version
Optimistic locking. Mỗi lần update, version tự tăng. Hai thread cùng update → một thành công, một fail (OptimisticLockException).
Thread A read v=1 → write v=2 OK
Thread B read v=1 → write v=2 FAIL! v đã là 2
@PrePersist / @PreUpdate
Lifecycle callbacks. @PrePersist chạy trước khi INSERT. @PreUpdate chạy trước khi UPDATE. Dùng để tự động set createdAt, updatedAt.
@PrePersist
protected void onCreate() {
    createdAt = LocalDateTime.now();
}

@PreUpdate
protected void onUpdate() {
    updatedAt = LocalDateTime.now();
}
@Enumerated
Cách lưu enum vào DB. ORDINAL = số (0, 1, 2...). STRING = tên ("PENDING", "CONFIRMED"...).
ORDINAL (cẩn thận!)
PENDING = 0
CONFIRMED = 1
Thêm enum ở giữa → số thay đổi → DB bị sai!
STRING (an toàn hơn)
"PENDING"
"CONFIRMED"
Thêm enum → không ảnh hưởng DB cũ

Java Class → Database Table Mapping

Java Entity
@Entity
@Table(name = "bookings")
public class Booking {

  @Id @GeneratedValue
  private Long id;

  @Column(nullable = false)
  private LocalDate bookingDate;

  @Enumerated(EnumType.STRING)
  private BookingStatus status;

  @Version
  private Long version;

  @ManyToOne
  private Service service;
}
Hibernate translates
→
SQL Table
CREATE TABLE bookings (
  id         BIGSERIAL    PRIMARY KEY,
  booking_date DATE       NOT NULL,
  status     VARCHAR(255),
  version    BIGINT,
  service_id BIGINT      REFERENCES services(id)
);
