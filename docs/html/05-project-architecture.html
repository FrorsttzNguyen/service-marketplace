Project Architecture — Kiến trúc dự án

Tài liệu này mô tả toàn bộ kiến trúc của dự án Service Marketplace: cấu trúc thư mục, phân tầng kiến trúc, cấu hình, Docker, Git workflow, chiến lược testing, và lộ trình phát triển. Đây là "bản đồ" để Hien hiểu project được tổ chức như thế nào và tại sao.

1. Dự án này là gì?

Service Marketplace — Nền tảng đặt dịch vụ đa nhà cung cấp

Service Marketplace là một nền tảng nơi:

  • Customer (khách hàng) — tìm và đặt dịch vụ (ví dụ: cắt tóc, vệ sinh nhà, sửa ống nước).
  • Vendor (nhà cung cấp) — đăng dịch vụ, quản lý lịch trình, nhận đặt chỗ.
  • Admin (quản trị) — quản lý nền tảng, kiểm duyệt, xem báo cáo.

Tương tự các nền tảng thực tế như Booksy, Thumbtack, hay TaskRabbit.

Tech Stack — Công nghệ sử dụng

Layer Technology Tại sao chọn?
Language Java 21 Industry standard, strong typing, ecosystem lớn, phù hợp cho enterprise backend.
Framework Spring Boot 3.5 Rapid development, auto-configuration, embedded server, cộng đồng lớn.
Database PostgreSQL 16 ACID-compliant, relational, free, phù hợp cho dữ liệu giao dịch tài chính.

Note

Đây là modular monolith, KHÔNG phải microservices. Mỗi domain module (user, vendor, booking, payment...) được phân tách rõ ràng trong cùng một ứng dụng. Thiết kế có đường dẫn rõ ràng để tách thành microservices sau này nếu cần.

Tại sao cấu trúc theo package thay vì theo layer?

Trong project này, cấu trúc ưu tiên theo layer (interfaces, application, domain, infrastructure), và bên trong mỗi layer chia theo domain module (user, vendor, booking...). Đây là cách làm của Layered Architecture kết hợp Domain-Driven Design.

Tip

Thư mục domain/ là trái tim của project. Mọi quy tắc nghiệp vụ (business rules) đều nằm ở đây. Nếu Hien muốn hiểu "hệ thống làm gì", hãy đọc code trong domain/ trước.

3. Kiến trúc phân tầng (Layered Architecture)

Nguyên tắc cốt lõi

Mỗi layer (tầng) có MỘT trách nhiệm duy nhất và chỉ phụ thuộc vào tầng bên dưới nó. Mũi tên dependency chỉ đi xuống, không bao giờ đi lên.

4 Tầng kiến trúc — Dependency Rule
INTERFACES LAYER
REST Controllers, DTOs
Nhận HTTP request, trả về JSON. KHÔNG chứa business logic.
↓
APPLICATION LAYER
Use Cases, Orchestration
Phối hợp giữa các domain object. KHÔNG chứa domain rules.
↓
DOMAIN LAYER (Core)
Entities, Value Objects, Enums
Business rules, state machines. KHÔNG phụ thuộc framework.
↓
INFRASTRUCTURE LAYER
Repositories, External Services
Database, Redis, Stripe. Implement domain interfaces.

Traoff: Domain layer dùng JPA annotations

Theo lý thuyết "Clean Architecture" thuần túy, domain layer không được phụ thuộc vào bất kỳ framework nào — kể cả JPA (@Entity, @Table...). Domain entity phải là POJO (Plain Old Java Object) thuần túy.

Nhưng trong project này, domain dùng JPA annotations. Tại sao?

Cách tiếp cận Ưu điểm Nhược điểm
Pure domain (không JPA)
Clean Architecture thuần		 Domain hoàn toàn độc lập. Dễ thay đổi database, framework. Phải viết mapping layer riêng (từ domain entity sang JPA entity). Nhiều class hơn, phức tạp hơn cho người mới học.
Domain + JPA annotations
Cách project này chọn		 Đơn giản, ít class, dễ hiểu cho người mới. Một entity vừa là domain model vừa là persistence model. Domain "biết" về JPA. Khó thay sang database NoSQL sau này. Vi phạm nguyên tắc Clean Architecture.

Traoff — Pragmatism over Purity

Trong thực tế, phần lớn Spring Boot project chọn cách thứ hai (domain + JPA annotations) vì đơn giản và thực tế hơn. Khi Hien đã thoải mái với kiến trúc này, có thể refactor sang Clean Architecture thuần trong project tiếp theo. Hiện tại: optimize cho việc học, không phải cho sự hoàn hảo lý thuyết.

Giai thích từng bước:

  1. BookingController (interfaces layer) — nhận JSON request, validate dữ liệu đầu vào (có đủ trường không? đúng format không?). Không chứa business logic.
  2. BookingService (application layer) — phối hợp (orchestrate): kiểm tra availability, tạo booking, gửi notification. Không chứa domain rules.
  3. Booking entity (domain layer) — chứa business rule: "chỉ book được nếu time slot còn trống". Đây là nơi logic nghiệp vụ sống.
  4. BookingRepository (infrastructure layer) — giao tiếp với PostgreSQL để lưu booking. Implement interface do domain layer định nghĩa.

4. Các file cấu hình

Mỗi file cấu hình có một vai trò cụ thể. Bảng dưới tóm tắt:

File Mục đích Cài đặt quan trọng
application.yml Cấu hình chung cho mọi môi trường jpa.ddl-auto: validate (không tự tạo table), flyway.enabled: true, server.port: 8080
application-dev.yml Cấu hình cho môi trường development (local) PostgreSQL trên localhost:5433, Redis trên localhost:6379, show-sql: true, log level DEBUG
application-test.yml Cấu hình cho chạy test H2 in-memory database (jdbc:h2:mem:testdb), Flyway disabled, security disabled
docker-compose.yml Định nghĩa Docker services cho development PostgreSQL 16 Alpine + Redis 7 Alpine, volume cho data persistence
.env.example Template cho biến môi trường (secrets) DB credentials, Stripe API keys, JWT secret. Copy thành .env và điền giá trị thực.
.gitignore Loại trừ file khỏi Git target/ (build artifacts), .env (secrets), IDE files (.idea/, .vscode/)

Bảo mật

File .env chứa secrets (database password, Stripe keys) và bị .gitignore loại trừ. Không bao giờ commit .env lên GitHub. File .env.example chỉ là template — không chứa giá trị thật.

Profile Override Mechanism

Spring Boot dùng profiles để override cấu hình. Khi chạy với profile dev, Spring đọc application.yml trước, rồi override bằng application-dev.yml:

Profile Override
application.yml
(base config)
+
application-dev.yml
(overrides)
=
Final Config
(effective config)

5. Môi trường phát triển (Docker Setup)

Tại sao dùng Docker?

Mỗi developer cần PostgreSQL và Redis để chạy project. Nếu mỗi người tự cài đặt:

  • Version khác nhau → bug khó reproduce.
  • Cài đặt trên macOS khác Linux khác Windows.
  • Cấu hình port, user, password có thể xung đột với service khác.

Docker giải quyết vấn đề này: một file docker-compose.yml duy nhất định nghĩa mọi thứ. Bất kỳ ai clone project đều có môi trường giống hệt nhau.

Why Docker uses 5433 — tránh xung đột port!

Note — Tại sao PostgreSQL chạy trên port 5433 thay vì 5432?

Port mặc định của PostgreSQL là 5432. Nếu máy đã cài PostgreSQL qua Homebrew, nó đang dùng port 5432. Docker map ra port 5433 để tránh xung đột. Spring Boot kết nối qua jdbc:postgresql://localhost:5433/marketplace.

Các lệnh Docker thường dùng

Lệnh Chức năng
docker compose up -d Khởi động PostgreSQL + Redis (chạy ngầm với flag -d = detached).
docker compose ps Kiểm tra trạng thái container (đang chạy hay stopped).
docker compose down Dừng container nhưng giữ nguyên data (trong Docker volume).
docker compose down -v Dừng container VÀ xóa data (xóa volume). Dùng khi muốn bắt đầu lại từ đầu.
docker compose logs -f postgres Xem log của PostgreSQL theo thời gian thực (-f = follow).
docker compose logs -f redis Xem log của Redis theo thời gian thực.

Conventional Commits

Mỗi commit message tuân theo định dạng chuẩn:

<type>(<scope>): <description>

# Ví dụ:
feat(booking): add time-slot conflict detection
fix(payment): handle duplicate webhook delivery
docs: update ERD diagram
Prefix Ý nghĩa Ví dụ
feat: Tính năng mới feat(booking): add time-slot conflict detection
fix: Sửa bug fix(payment): handle duplicate webhook delivery
docs: Tài liệu docs: update ERD diagram
refactor: Tái cấu trúc code (không đổi behavior) refactor(domain): extract Money value object
test: Thêm/cập nhật test test(booking): add concurrency test
chore: Build, config, dependencies chore(docker): upgrade PostgreSQL to 17

Note

Commit message viết bằng English, imperative mood (thức mệnh lệnh). Viết "add" thay vì "added", "fix" thay vì "fixed". Nghĩ như: "Nếu merge commit này, nó sẽ add / fix cái gì?"

7. Chiến lược Testing

Testing Pyramid

Chiến lược test của project theo mô hình Testing Pyramid — nhiều unit test ở đáy, ít E2E test ở trên cùng:

Testing Pyramid
E2E Tests
Iít, đắt, test toàn flow
Integration Tests
Trung bình, test với real DB (TestContainers)
Unit Tests
Nhiều, nhanh, test business logic (Money, BookingStatus, PricingType)

Nội dung test theo từng loại

Loại test Test cái gì? Tool
Unit Tests Money arithmetic (cộng, trừ, nhân, chia),
BookingStatus transitions (PENDING → CONFIRMED → COMPLETED),
PricingType calculations (FIXED vs HOURLY pricing) 		JUnit 5, AssertJ
Repository Tests CRUD operations, custom queries (tìm booking theo vendor, theo date range) Spring Data JPA Test, H2 hoặc TestContainers
API Tests Endpoint responses, authentication, validation (sai input → 400, không token → 401) MockMvc, WebMvcTest
Integration Tests Payment webhook flow, booking concurrency (2 customer đặt cùng slot), transaction rollback TestContainers (PostgreSQL thật trong Docker), Stripe test mode

Mục tiêu Coverage

80%+ coverage trên service/domain layer. Không bắt buộc 100% — tập trung vào những nơi chứa business logic quan trọng nhất.

Tại sao dùng TestContainers thay vì H2 cho integration test?

H2 là in-memory database, nhanh nhưng không giống hoàn toàn PostgreSQL. Một số query hoạt động trên H2 nhưng thất bại trên PostgreSQL (ví dụ: type casting, array operations). TestContainers chạy PostgreSQL thật trong Docker container — test trung thực hơn.

10. Tóm tắt

Key Takeaways

  • Layered Architecture: interfaces → application → domain → infrastructure. Mỗi tầng một trách nhiệm, dependency chỉ đi xuống.
  • Domain là core: mọi business rule sống ở đây. Infrastructure chỉ là adapter kết nối hệ thống bên ngoài.
  • Domain + JPA annotations: chọn sự đơn giản thay vì Clean Architecture thuần túy — pragmatic choice cho người mới học.
  • Docker: môi trường phát triển đồng nhất. Mọi người cùng PostgreSQL + Redis version, không xung đột port.
  • Git flow: feature branch + PR + conventional commits = lịch sử code sạch, dễ theo dõi.
  • Testing pyramid: nhiều unit test (nhanh), ít integration test (chậm nhưng chính xác), rất ít E2E test (đắt nhất).
  • 8 phases: xây dựng từng bước, mỗi phase xây lên phase trước, Hien học dần concepts qua thực hành.
