1. Database là gì? Tại sao cần?

💡 Đơn giản: Database = nơi lưu trữ dữ liệu có cấu trúc, giống một tủ hồ sơ siêu tốc. Thay vì ghi ra file text rồi tự tìm kiếm, database giúp bạn lưu, tìm, sửa, xóa dữ liệu cực nhanh.

2. Relational Database (Cơ sở dữ liệu quan hệ)

Relational DB = dữ liệu được tổ chức thành bảng (table), các bảng liên kết nhau qua khóa (keys).

Khái niệm cơ bản

Thuật ngữGiải thíchVí dụ
Table (Bảng)Tập hợp dữ liệu cùng loạiusers, bookings
Row (Dòng)Một bản ghi dữ liệuUser: Hien, hien@email.com, CUSTOMER
Column (Cột)Một thuộc tính của dữ liệuemail, full_name, role
Primary Key (Khóa chính)Định danh DUY NHẤT mỗi dòngid = 1, id = 2, ...
Foreign Key (Khóa ngoại)Liên kết đến bảng khácbookings.vendor_id → vendors.id

🎯 ACID = why it matters:
Atomicity — Transaction要么全 thành công要么全 thất bại (không có nửa vời)
Consistency — Dữ liệu luôn ở trạng thái hợp lệ (constraint được enforce)
Isolation — 2 transaction chạy song song không ảnh hưởng nhau
Durability — Khi đã commit, dữ liệu không bị mất dù server crash

6. Money as Cents — Tại sao KHÔNG dùng FLOAT?

⚠️ Đây là bug cực kỳ phổ biến: Dùng float hoặc double cho tiền → sai số làm mất tiền thật!

Ví dụ lỗi float:

// ❌ SAI — floating point precision issues
double price = 19.99;
double total = price * 100;
System.out.println(total);
// Output: 1998.9999999999998 (NOT 1999!)

// Trong database:
double balance = 0.1 + 0.2;
System.out.println(balance);
// Output: 0.30000000000000004 (NOT 0.3!)

Cách đúng — integer cents:

// ✅ ĐÚNG — integer cents, zero precision loss
long priceCents = 1999;  // $19.99 = 1999 cents
long totalCents = priceCents * 100;
System.out.println(totalCents);
// Output: 199900 (exact!)

// Trong database:
CREATE TABLE services (
    base_price_cents BIGINT NOT NULL CHECK (base_price_cents > 0)
    // BIGINT = max 9,223,372,036,854,775,807 cents
    // = ~92 trillion dollars — đủ rồi!
);
CáchPrecision$19.99 lưu nhưRisk
FLOAT~7 digits19.9899997711...❌ Sai số
DOUBLE~15 digits19.98999999999998⚠️ Vẫn sai
DECIMAL(10,2)Exact19.99⚠️ Chậm hơn integer
BIGINT (cents)Exact1999✅ Nhanh, chính xác

7. Optimistic Locking — Ngăn double-booking

_race condition là gì?

Khi 2 người cùng đặt 1 time slot CÙNG LÚC, hệ thống cần đảm bảo chỉ 1 người đặt được.

Thread A ✅

READ booking (version=1)
    ↓
CHECK: slot available? YES
    ↓
WRITE booking (version → 2)
    ↓
✅ SUCCESS

Thread B ❌

READ booking (version=1)
    ↓
CHECK: slot available? YES
(stale data!)
    ↓
WRITE booking (version → 2)
    ↓
❌ OptimisticLockException!
(retry or 409 Conflict)

2 lớp bảo vệ

LớpCơ chếCode
Lớp 1: Application JPA @Version — Hibernate tự động check version khi UPDATE 
@Entity
public class Booking {
    @Version
    private Long version;
    // UPDATE WHERE id=? AND version=?
    // Nếu version đã thay đổi → exception
}
Lớp 2: Database UNIQUE constraint — ngay cả khi optimistic lock fail, DB vẫn chặn 
-- Safety net
ALTER TABLE bookings
ADD CONSTRAINT uq_booking_slot
UNIQUE (service_id, booking_date, start_time);
🎯 Tại sao optimistic thay vì pessimistic?
Pessimistic locking (SELECT ... FOR UPDATE) lock row → thread khác PHẢI CHỜ. Chậm hơn.
Optimistic locking: ai cũng đọc được, chỉ conflict khi ghi → nhanh hơn ở đa số trường hợp (conflict hiếm).
Với booking service (conflict rate thấp), optimistic là chọn đúng. Xem ADR-0003.

8. Flyway — Quản lý Database Schema

Vấn đề không có migrations:

❌ Không có Flyway
• Chạy SQL thủ công → quên chạy trên production
• Team conflict — ai cũng sửa DB
• Không biết DB đang ở version nào
• Không rollback được
✅ Có Flyway
• SQL versioned trong code → review được qua PR
• Chạy tự động khi app start
• Biết chính xác DB ở version nào
• Có thể rollback (với Flyway Pro)

Flyway chạy theo thứ tự V1 → V2 → ... → V6. Mỗi file chỉ chạy 1 lần, tracked trong bảng flyway_schema_history

Ví dụ: V1 migration

-- V1__create_users_table.sql
-- Tạo bảng users và refresh_tokens cho authentication

CREATE TABLE users (
    id          BIGSERIAL           PRIMARY KEY,
    email       VARCHAR(255)       NOT NULL UNIQUE,    -- UNIQUE: không được trùng email
    password_hash VARCHAR(255)    NOT NULL,              -- BCrypt hash, không lưu plaintext!
    full_name   VARCHAR(255)       NOT NULL,
    phone       VARCHAR(20),
    role        VARCHAR(20)        NOT NULL
                CHECK (role IN ('CUSTOMER', 'VENDOR', 'ADMIN')),  -- Enum constraint
    status      VARCHAR(20)        NOT NULL DEFAULT 'ACTIVE',
    created_at  TIMESTAMP           NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMP           NOT NULL DEFAULT NOW()
);

CREATE TABLE refresh_tokens (
    id          BIGSERIAL           PRIMARY KEY,
    user_id     BIGINT              NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token       VARCHAR(500)       NOT NULL,
    expires_at  TIMESTAMP           NOT NULL
);

-- Index: tìm user theo email nhanh (dùng khi login)
CREATE INDEX idx_users_email ON users(email);

9. Indexes — Tại sao cần?

💡 Analogy: Index = mục lục trong sách. Không có mục lục → phải đọc từng trang. Có mục lục → mở đúng trang ngay.

10. Constraints — Ràng buộc dữ liệu

ConstraintÝ nghĩaVí dụ trong project
NOT NULL Field bắt buộc — không được để trống email NOT NULL — mọi user phải có email
UNIQUE Giá trị không được trùng trong toàn bảng email UNIQUE — 2 user không cùng email
CHECK Validate giá trị phải thỏa mãn điều kiện CHECK (rating BETWEEN 1 AND 5)
CHECK (base_price_cents > 0)
CHECK (role IN ('CUSTOMER','VENDOR','ADMIN'))
FOREIGN KEY Giá trị phải tồn tại ở bảng khác. Đảm bảo tính toàn vẹn bookings.service_id REFERENCES services(id)
→ Không thể tạo booking cho service không tồn tại
ON DELETE CASCADE Xóa parent → tự động xóa children refresh_tokens.user_id REFERENCES users(id) ON DELETE CASCADE
→ Xóa user → refresh tokens cũng bị xóa theo

Khi constraint bị vi phạm:

-- Thử insert 2 user cùng email
INSERT INTO users (email, ...) VALUES ('hien@email.com', ...);
-- ✅ OK

INSERT INTO users (email, ...) VALUES ('hien@email.com', ...);
-- ❌ ERROR: duplicate key value violates unique constraint "users_email_key"

11. Tóm tắt

🎯 Key Takeaways:
  • 17 bảng trong database, tổ chức thành 5 nhóm: Users, Vendors, Services, Bookings, Orders/Payments
  • Money as cents (BIGINT) — không bao giờ dùng float cho tiền
  • Optimistic locking + UNIQUE constraint = 2 lớp chống double-booking
  • Flyway quản lý schema qua versioned SQL files
  • Indexes trên các cột tìm kiếm nhiều (email, booking slot, Stripe IDs)
  • Constraints (NOT NULL, UNIQUE, CHECK, FK) bảo vệ data integrity ở database level
