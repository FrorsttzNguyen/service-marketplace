System Design — Thiết kế Hệ thống

Trang này giải thích toàn bộ system design (thiết kế hệ thống) của project Service Marketplace. Hien sẽ học cách tư duy về kiến trúc phần mềm — từ yêu cầu, đến diagram, đến quyết định kỹ thuật — thông qua một project thực tế.

Tại sao System Design quan trọng?

Trong phỏng vấn mid/senior, câu hỏi system design gần như chắc chắn xuất hiện. Hơn thế, hiểu system design giúp Hien viết code có cấu trúc tốt hơn ngay từ đầu, thay vì refactor loạn xà sau này.

1. System Design là gì?

System Design là nghệ thuật thiết kế hệ thống phần mềm trước khi code. Giống như xây nhà — Hien cần bản vẽ kiến trúc trước khi đổ bê tông. Nếu bỏ qua bước này, code sẽ trở thành "mì spaghetti" — mọi thứ dính vào nhau, sửa một chỗ hỏng ba chỗ khác.

2. Requirements — Project này cần gì?

Trước khi thiết kế kiến trúc, cần nêu rõ hệ thống phải làm được gì (functional) và làm tốt như thế nào (non-functional).

Functional Requirements (Chức năng)

Những thứ hệ thống phải làm được:

STT Chức năng Mô tả
1 Authentication & Authorization Đăng ký, đăng nhập với 3 roles: Customer, Vendor, Admin. JWT-based.
2 Vendor Onboarding Vendor đăng ký, upload giấy tờ, Admin xác minh (verification).
3 Service CRUD Vendor tạo/sửa/xóa dịch vụ. Bao gồm category, hình ảnh, giá, thời lượng.
4 Booking & Time Slots Customer chọn dịch vụ, chọn ngày giờ, tạo booking. Quản lý slot tránh double-booking.
5 Payment (Stripe) Thanh toán qua Stripe. Webhook xác nhận. Idempotent.
6 Review & Rating Customer đánh giá dịch vụ sau khi hoàn thành. 1-5 sao + comment.
7 Notifications Email thông báo khi booking confirmed, payment completed, v.v.
8 Admin Moderation Admin duyệt Vendor, kiểm soát nội dung, xem thống kê.

Non-Functional Requirements (Phi chức năng)

Những đặc tính chất lượng của hệ thống — không phải tính năng, nhưng quyết định hệ thống có dùng được không:

Đặc tính Yêu cầu Giải pháp kỹ thuật
Consistency Không bao giờ double-booking — một slot chỉ được book bởi 1 người Optimistic locking + UNIQUE constraint
Idempotency Payment webhook xử lý chính xác 1 lần, dù Stripe gửi nhiều lần stripe_event_log table + unique constraint trên event_id
Performance Search < 200ms Redis cache (cache-aside pattern)
Security Chỉ người đúng role mới truy cập đúng resource JWT auth + RBAC (Role-Based Access Control) + input validation
Auditability Mọi giao dịch tài chính phải có log Audit log table, không bao giờ DELETE financial records

Functional vs Non-Functional — cách nhớ

Functional = "Hệ thống LÀM GÌ?" (đặt lịch, thanh toán, đăng nhập...)
Non-Functional = "Hệ thống LÀM TỐT KHÔNG?" (nhanh không, an toàn không, đúng không...)

Tại sao chọn Modular Monolith?

Lý do Giải thích
Single developer Một người làm — không cần independent deployment như microservices (nhiều team song song).
Sequential phases Phát triển theo từng phase tuần tự — không cần nhiều team chạy song song.

Lưu ý quan trọng

"Modular Monolith" KHÔNG phải là "big ball of mud" (code lộn xộn). Mỗi module có boundary rõ ràng — interfaces riêng, domain riêng. Điểm khác biệt duy nhất là tất cả chạy trong một process, một database.

5. Request Flow — Customer đặt dịch vụ

Đây là flow hoàn chỉnh khi Customer tìm kiếm dịch vụ, đặt lịch, và thanh toán. Đọc từng bước, hiểu data đi từ đâu đến đâu.

Phase A: Tìm kiếm dịch vụ (có cache)

Search Flow — Cache-Aside Pattern
1Customer → API: GET /api/services?category=spa
2API → Redis: check cache với key services:category:spa
3Redis → API: CACHE MISS (lần đầu, chưa có data)
4API → PostgreSQL: SELECT * FROM services WHERE category='spa'
5PostgreSQL → API: danh sách services
6API → Redis: write cache (TTL 5 phút)
7API → Customer: JSON danh sách services

Lần 2 request giống nhau

Step 3 sẽ là CACHE HIT — Redis trả data luôn, KHÔNG cần query PostgreSQL. Nhanh hơn  ~50x (1ms vs 50ms).

Phase B: Tạo booking

Booking Flow — Optimistic Locking
8Customer → API: POST /api/bookings {serviceId, date, time}
9API → PostgreSQL: INSERT booking (version=1)
10PostgreSQL → API: OK — UNIQUE constraint đảm bảo không double-booking
11API → Customer: booking confirmed + order created (status: PENDING)

Phase C: Thanh toán

Payment Flow — Stripe + Webhook
12Customer → API: POST /api/orders/{id}/pay
13API → Stripe: create PaymentIntent
14Stripe → API: trả về client_secret
15API → Customer: redirect URL to Stripe Checkout
16Customer → Stripe: nhập thẻ, thanh toán
17Stripe → API: webhook payment_intent.succeeded
18API → PostgreSQL: INSERT payment record + UPDATE order status → PAID
19API → Stripe: 200 OK (webhook acknowledged)

6. Payment Flow — Stripe Integration

Phần này đi sâu hơn vào flow thanh toán — đặc biệt là webhook idempotency, một chủ đề rất quan trọng trong system design.

Sequence Diagram — Stripe Payment với Webhook

Payment + Webhook Idempotency Flow
Customer
API Server
Stripe
Database
1Customer → API: POST /pay
2API → Stripe: Create PaymentIntent
3Stripe → API: client_secret
4API → Customer: Checkout URL

6Stripe → API: Webhook payment_intent.succeeded

8API → DB: Check stripe_event_log — event_id đã tồn tại chưa?
9DB → API: Not found (lần đầu — tiếp tục xử lý)
10API → DB: INSERT payment record + INSERT stripe_event_log
11API → DB: UPDATE order → status = PAID
12API → Stripe: 200 OK

Webhook Idempotency — Tại sao quan trọng?

Vấn đề: Stripe có thể gửi cùng một webhook NHIỀU LẦN

Network không đáng tin cậy. Stripe gửi webhook, server nhận được nhưng response bị mất → Stripe nghĩ là fail → gửi lại. Nếu API không handle đúng, customer bị charge 2 lần hoặc order bị cập nhật sai.

Giải pháp: Bảng stripe_event_log ghi lại mọi event đã xử lý.

// Pseudocode — Webhook handler
fun handleWebhook(event: StripeEvent) {
    // 1. Verify signature — đảm bảo từ Stripe thật
    if (!verifySignature(event)) throw UnauthorizedException()

    // 2. Check idempotency — đã xử lý chưa?
    val eventId = event.id
    if (eventLogRepository.existsByEventId(eventId)) {
        // Đã xử lý rồi → return OK, KHÔNG xử lý lại
        return ResponseEntity.ok()
    }

    // 3. Xử lý lần đầu
    val tx = transactionTemplate.execute {
        paymentRepository.save(payment)       // INSERT payment
        eventLogRepository.save(eventLog)      // INSERT event log
        orderRepository.updateStatus(PAID)     // UPDATE order
    }
}

Key insight: Bước kiểm tra + insert diễn ra trong cùng 1 database transaction. UNIQUE constraint trên event_id đảm bảo ngay cả trong race condition, chỉ 1 thread insert được — thread kia sẽ bị constraint violation.

7. Booking Concurrency — Optimistic Locking

Đây là vấn đề kinh điển trong hệ thống đặt lịch: hai người cùng đặt 1 slot cùng lúc. Nếu không xử lý đúng → double-booking.

Race Condition — Visual Timeline

Race Condition — Hai Thread cùng đặt 1 slot

Thread A

READ booking (version=1)
CHECK: slot available?
WRITE booking (v→2)
✓ SUCCESS — version khớp, insert OK

Thread B

READ booking (version=1)
CHECK: slot available?
...đang chờ...
✗ FAIL — version không khớp (đã là 2, expect 1)
→ OptimisticLockException!
Retry HOẶC return 409 Conflict

Hai lớp bảo vệ

Project này dùng 2 lớp để chống double-booking — phòng thủ sâu (defense in depth):

Lớp 1: Optimistic Locking (Application level)

@Entity
class Booking {
    @Version
    private Long version;  // JPA tự động quản lý

    // Khi UPDATE, JPA tự thêm: WHERE version = expected_version
    // Nếu 0 rows affected → OptimisticLockException
}

Lớp 2: UNIQUE Constraint (Database level)

-- Ngay cả khi optimistic lock bị bypass somehow,
-- database vẫn từ chối duplicate
ALTER TABLE bookings
  ADD CONSTRAINT unique_booking_slot
  UNIQUE (service_id, booking_date, start_time);

Tại sao dùng 2 lớp?

Optimistic locking cho phép retry logic đẹp (application xử lý). UNIQUE constraint là safety net cuối cùng — database-level guarantee. Nếu một ngày nào đó code bị bug bỏ qua optimistic lock, DB vẫn bảo vệ data integrity.

8. Caching Strategy — Redis Cache-Aside

Caching = lưu data đã query vào memory (Redis), lần sau đọc nhanh hơn không cần hit database. Pattern dùng trong project này là Cache-Aside (Lazy Loading).

Tại sao không cache "mãi mãi"?

TTL (Time To Live) đảm bảo data cũ tự động hết hạn. Nếu Vendor cập nhật giá service nhưng Hien quên invalidate cache → TTL sẽ tự xóa sau vài phút. Đây là cơ chế bảo vệ thứ hai.

9. Security — JWT Authentication

JWT (JSON Web Token) là cách phổ biến nhất để authenticate REST API. Khác với session-based auth (server lưu state), JWT là stateless — server không cần lưu token nào cả.

JWT Authentication Flow

JWT Flow — Login, Access, Refresh, Logout

Bước 1: Login

1Client → POST /api/auth/login {email, password}
2Server verify password (BCrypt hash match)
3Server → Client: {accessToken: "eyJ...", refreshToken: "eyJ..."}

Bước 2: Access protected resource

4Client → GET /api/bookings
Header: Authorization: Bearer <accessToken>
5Server validate JWT signature + check expiry
6Server check role (CUSTOMER ✓ / ANONYMOUS ✗)
7Server → Client: booking data

Bước 3: Token Refresh

8Client → POST /api/auth/refresh {refreshToken}
9Server validate refresh token → generate new accessToken
10Server → Client: {accessToken: "new_eyJ..."}

Bước 4: Logout

11Client → POST /api/auth/logout
12Server blacklist refresh token trong Redis (với TTL = token expiry)
13Client xóa local tokens → logged out

Tại sao JWT thay vì Session?

Tiêu chí JWT (dùng trong project) Session
State Stateless — server không lưu gì Stateful — server lưu session trong memory/DB
Scale Dễ horizontal scale (bất kỳ server nào verify được) Cần sticky session hoặc session store chung
REST API fit Phù hợp — REST là stateless Không phù hợp lắm — REST nên stateless
Mobile friendly Token lưu client — mobile app dễ dùng Cookie-based — mobile phức tạp hơn
Revocation Khó hơn — cần blacklist (dùng Redis) Dễ — xóa session server-side

Access Token vs Refresh Token

Access Token: short-lived (15-30 phút). Dùng cho mọi API call. Nếu bị lộ, hacker chỉ dùng được trong thời gian ngắn.
Refresh Token: long-lived (7-30 ngày). Chỉ dùng để lấy access token mới. Lưu ở Redis nên có thể revoke bất cứ lúc nào.

10. Scale Considerations

Một câu hỏi phỏng vấn phổ biến: "Hệ thống này scale như thế nào?". Đây là câu trả lời — thể hiện tư duy thực tế, không phải "chạy theo buzzword".

Scale Levels

Scale
Architecture thay đổi thế nào
Current
1 user
1 server, 1 database. Monolith là đủ. Redis caching optional. Đúng với project này.
10x
~100 users
Thêm Redis caching cho read-heavy queries. Thêm PostgreSQL read replica để giảm load cho primary DB. Horizontal scaling với load balancer (2-3 API instances).
100x
~10K users
Extract Notification Service thành service riêng + message queue (RabbitMQ/Kafka). Consider CQRS — tách read model và write model. Connection pooling tối ưu.
1000x
~1M+ users
Chuyển sang Microservices. Database sharding. CDN cho images. Separate search engine (Elasticsearch). Rate limiting, API gateway.

Điểm nhấn cho phỏng vấn

Khi trả lời câu hỏi scale, KHÔNG nói "em sẽ dùng microservices". Hãy nói: "Với scale hiện tại, monolith là lựa chọn đúng. Khi traffic tăng 10x, em sẽ thêm caching và read replica. Chỉ khi cần mới chuyển sang microservices." Câu trả lời này thể hiện judgment tốt hơn rất nhiều.

11. ADR Summaries

ADR (Architecture Decision Record) là tài liệu ghi lại mỗi quyết định kiến trúc quan trọng — gồm Context (tình huống), Decision (quyết định), Consequences (hệ quả). Dưới đây là tóm tắt 6 ADR của project.

ADR-1: Chọn PostgreSQL thay vì MongoDB

PostgreSQL

Context: Cần chọn database. Dữ liệu có quan hệ rõ ràng (User → Booking → Payment), cần transaction đảm bảo consistency.

Decision: PostgreSQL — relational database, ACID transactions, mature ecosystem.

Lý do: Booking/Payment cần strong consistency. MongoDB không hỗ trợ transaction multi-document tốt bằng (trước version 4). Project học tập nên chọn phổ biến nhất trong enterprise.

ADR-2: Redis Cache-Aside Pattern

Cache-Aside

Context: Search services là read-heavy. Query DB mỗi lần → chậm khi scale.

Decision: Cache-Aside với Redis. Application tự quản lý cache (read cache first, miss → read DB, write cache). Invalidate trên write.

Lý do: Đơn giản, dễ hiểu, dễ debug. Write-through/Write-behind phức tạp hơn mà không cần thiết ở scale này.

ADR-3: Optimistic Locking cho Bookings

Optimistic Locking

Context: Hai customer có thể đặt cùng 1 slot cùng lúc. Cần ngăn double-booking.

Decision: JPA @Version (optimistic locking) + UNIQUE constraint database level.

Lý do: Optimistic locking không lock DB row → performance tốt hơn pessimistic locking. Conflict hiếm (không phải booking concert). UNIQUE constraint là safety net.

ADR-4: Stripe Webhook Idempotency

Event Log Table

Context: Stripe gửi webhook, network không đáng tin cậy → có thể gửi lại cùng event. Nếu xử lý 2 lần → double charge hoặc sai status.

Decision: Bảng stripe_event_log với UNIQUE constraint trên event_id. Check + INSERT trong cùng transaction.

Lý do: Database-level guarantee mạnh nhất. Không phụ thuộc application logic. Dễ test, dễ verify.

ADR-5: Modular Monolith thay vì Microservices

Modular Monolith

Context: Cần chọn kiến trúc cho project. Single developer, learning vehicle, small scale.

Decision: Modular Monolith — module boundaries rõ ràng nhưng chạy trong 1 process, 1 database.

Lý do: Tránh distributed systems complexity (network calls, data consistency, deployment orchestration). Module boundaries giúp extract thành microservices sau này nếu cần.

ADR-6: JWT thay vì Session Auth

JWT (Stateless)

Context: Cần chọn authentication mechanism cho REST API.

Decision: JWT với access token (short-lived) + refresh token (long-lived, stored in Redis).

Lý do: REST API nên stateless — JWT fit hơn session. Dễ scale horizontally. Refresh token blacklist qua Redis cho phép revocation khi cần.

12. Tóm tắt

Key Takeaways

Những điều cần nhớ cho phỏng vấn

  • Luôn bắt đầu bằng requirements — functional + non-functional — trước khi nhảy vào kiến trúc.
  • Modular Monolith là lựa chọn hợp lý cho small team / single developer. Microservices là khi CẦN, không phải khi THÍCH.
  • Defense in depth — optimistic locking + UNIQUE constraint = 2 lớp bảo vệ cho critical data.
  • Idempotency — mọi webhook / external event PHẢI xử lý idempotent. Database unique constraint là cách đáng tin cậy nhất.
  • Cache-Aside — đơn giản, hiệu quả. Luôn set TTL. Invalidate trên write.
  • JWT stateless — phù hợp REST API, dễ scale. Refresh token + blacklist cho revocation.
  • Scale theo nhu cầu — không premature optimization. Thêm caching → read replica → message queue → microservices.

Câu hỏi luyện tập

Những câu hỏi này giúp Hien chuẩn bị cho phỏng vấn system design. Thử trả lời bằng miệng trước khi xem gợi ý:

# Câu hỏi Gợi ý trả lời
1 Thiết kế hệ thống đặt lịch (booking system). Cách chống double-booking? Optimistic locking + UNIQUE constraint. Explain race condition.
2 Làm sao đảm bảo payment webhook chỉ xử lý 1 lần? Idempotency key / event log table. Check + INSERT trong 1 transaction.
3 Khi nào nên chọn monolith, khi nào microservices? Monolith khi small team / early stage. Microservices khi multiple teams / independent scale / >1000x.
4 Cache strategy cho read-heavy system? Cache-Aside. TTL. Invalidation on write. Mention cache stampede.
5 JWT vs Session — tradeoffs? JWT = stateless, scalable, harder to revoke. Session = stateful, easy revoke, harder to scale.
6 Hệ thống này scale từ 100 users lên 1M users, thay đổi gì? Caching → read replicas → message queue → microservices → sharding → CDN.
